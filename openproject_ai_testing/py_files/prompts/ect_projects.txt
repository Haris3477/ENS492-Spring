You are a software tester performing Equivalence Class Testing (ECT) on the OpenProject REST API.

Target endpoint: Projects
Base URL: {BASE_URL}
Authentication: Basic auth with username "apikey" and password "{API_KEY}"

The Projects API works as follows:
- GET    /api/v3/projects           -> list all projects
- POST   /api/v3/projects           -> create a project
- GET    /api/v3/projects/{id}      -> get a project
- PATCH  /api/v3/projects/{id}      -> update a project
- DELETE /api/v3/projects/{id}      -> delete a project

When creating or updating a project, the request body must be JSON with these fields:
- "name": string (required, must not be empty, maximum 255 characters)
- "identifier": string (required, must be unique, lowercase letters/numbers/hyphens only, maximum 100 characters)
- "public": boolean (optional)
- "description": object with "raw" string field (optional)

Equivalence Class Testing means you divide inputs into classes where all values in a class are expected to behave the same way, then test one representative value from each class:

For the "name" field:
- Valid class: a normal string between 1 and 255 characters (should succeed with 201)
- Invalid class 1: empty string (should fail with 422)
- Invalid class 2: string longer than 255 characters (should fail with 422)

For the "identifier" field:
- Valid class 1: lowercase letters only (should succeed with 201)
- Valid class 2: lowercase letters with hyphens (should succeed with 201)
- Valid class 3: lowercase letters with numbers (should succeed with 201)
- Invalid class 1: empty string (should fail with 422)
- Invalid class 2: contains uppercase letters (should fail with 422)
- Invalid class 3: contains spaces (should fail with 422)
- Invalid class 4: contains special characters other than hyphens e.g. underscore or @ (should fail with 422)

For the "public" field:
- Valid class 1: true (should succeed with 201)
- Valid class 2: false (should succeed with 201)

For missing required fields:
- Missing name entirely (should fail with 422)
- Missing identifier entirely (should fail with 422)

Write a complete Python test script that:
1. Uses the requests library with Basic auth
2. Sets credentials like this exactly:
   import os
   API_KEY = os.environ.get("OP_API_KEY")
   BASE_URL = os.environ.get("OP_BASE_URL", "http://localhost:8080")
   auth = ("apikey", API_KEY)
3. Tests all the equivalence classes listed above
4. Always cleans up any successfully created projects by deleting them after testing
5. Prints PASS or FAIL for each test case with a short description and the actual status code on FAIL
6. Generates unique identifiers using both timestamp AND 4 random characters like this:
   import time, random, string
   def unique_id():
       suffix = ''.join(random.choices(string.ascii_lowercase, k=4))

7. Never hardcodes credentials anywhere in the code
8. Handles all HTTP errors gracefully without crashing

Coverage instrumentation (REQUIRED for every assertion — independent line counting):
For each test case you generate, measure how many Ruby lines that case exercises *independently* (so two cases hitting the same code path each get full credit):
1. Reset coverage state immediately before the assertion's HTTP call:
   requests.get(f"{BASE_URL}/__cov_reset__")
2. Read the count after the reset (should be ~0):
   cov_before = int(requests.get(f"{BASE_URL}/__cov_count__").text)
3. Make the request and check the status as you would normally.
4. Read the count after the response:
   cov_after = int(requests.get(f"{BASE_URL}/__cov_count__").text)
5. Compute delta = cov_after - cov_before
6. Append "[lines={delta}]" to the PASS or FAIL message.

Example pattern:
    requests.get(f"{BASE_URL}/__cov_reset__")

Rules:
- Every PASS or FAIL line MUST end with the [lines=N] marker.
